# Definição de Pronto (DoD) — Lab 7

## DoD — Requisito (REQ-###) -

Um requisito é considerado "Pronto" quando:

1. `REQ-###` tem um ID estável, título claro e tipo explicitamente identificado (`FR`, `NFR` ou restrição)
2. O stakeholder e a source/origem do requisito estão registados
3. A descrição é não ambígua e evita detalhar implementação desnecessariamente
4. Existem acceptance criteria em número suficiente (`2–4`, quando aplicável) e são testáveis ou verificáveis
5. O impacto da variante está explicitamente identificado (`Yes/No`) quando aplicável
6. Existem ligações a use cases ou user stories relacionados, quando disponíveis
7. Conflitos, duplicados ou problemas de scope estão resolvidos ou documentados
8. Está definido um método de validação (revisão, demo, teste ou medição)

## DoD — User Story (US-###)

Uma user story é considerada "Pronta" quando:

1. A user story indica claramente ator, necessidade e valor
2. O âmbito é suficientemente pequeno para ser compreendido e validado
3. Os acceptance criteria estão acordados e são verificáveis
4. A user story está ligada à feature e ao(s) requisito(s) relevantes
5. Quaisquer restrições específicas da variante estão refletidas quando aplicável
6. A evidência esperada de conclusão é conhecida (demo, teste, revisão ou medição)
7. Não existe ambiguidade por resolver que bloqueie a validação da user story
8. Os stakeholders conseguem decidir objetivamente se a user story está completa
