# Lab 11 — Test-First Vibe Coding:

## <mark style="color:blue;">Write Tests Before Code (PyTest/JUnit) + BDD Scenarios (Cucumber/Behave) + Optional UI (Selenium</mark><mark style="color:blue;">)</mark>

**Tools (choose what fits your stack):**

- **PyTest** _(Python)_ or **JUnit** _(Java)_
- **Cucumber** _(Java)_ or **Behave** _(Python)_ for BDD/Gherkin
- **Selenium** _(optional, UI-only and only if your project has a stable web UI)_

**Input:** `docs/requirements_v1.md`, `docs/acceptance_criteria.md`, `bdd/features/lab9.feature`, `docs/test_plan.md`\
**Output:** tests in code + updated feature file + `docs/test_first_log.md`

This lab applies a test-first approach: you will define behavior via tests **before** writing (or generating) implementation code.\
You may use AI tools to help implement code, but the contract is the test suite.

> **GitHub rule:** All deliverables must be committed to your **team GitHub repository** under the exact paths listed below.\
> If it is not in the repo, it is not considered delivered.

---

### <mark style="color:blue;">Objectives</mark>

By the end of Lab 11, your team should:

- Implement a small slice using **tests first** (TDD mindset)
- Create automated tests that verify AC/behavior
- Produce BDD scenarios that align with requirements (and optionally run them)
- Demonstrate at least one negative/error test and one boundary test
- Keep a short log of test-first iterations and decisions

---

### <mark style="color:blue;">Scope rule</mark>

Pick **one slice** (small and coherent) and implement only:

- **3 requirements** (REQ-###) maximum
- plus their key acceptance criteria

Choose requirements that are suitable for testing:

- validation rules
- business rules
- data transformations
- small service functions

> Avoid trying to automate a full UI end-to-end flow unless the UI is stable and you truly need Selenium.

---

### <mark style="color:blue;">Recommended repo structure</mark>

Use the structure from earlier labs:

- `/tests` for unit/integration tests
- `/bdd` for BDD tooling (optional)
- `/docs` for traceability and logs

Examples:

- Python:
  - `tests/test_<module>.py`
  - `bdd/features/`
  - `bdd/steps/`
- Java:
  - `src/test/java/...`
  - `src/test/resources/features/`

---

### <mark style="color:blue;">In-class tasks (step by step)</mark>

#### <mark style="color:$primary;">1) Select your 3 REQs + acceptance criteria</mark>

From `docs/requirements_v1.md` and `docs/acceptance_criteria.md`, select:

- REQ-A, REQ-B, REQ-C (max 3)

For each selected REQ:

- list the AC you intend to automate

Record this selection in `docs/test_first_log.md`.

---

#### <mark style="color:$primary;">2) Write tests first (PyTest or JUnit)</mark>

Create automated tests **before** writing implementation.

Minimum test set:

- **6 tests total**
  - at least **3 happy path** assertions
  - at least **2 negative/error** tests
  - at least **1 boundary** test

**Rule:** each test must clearly map to an AC item.

---

#### <mark style="color:$primary;">3) Implement the minimum code to pass (TDD)</mark>

Now implement the minimal code to satisfy tests:

- do not add extra features
- keep scope limited to the selected REQs

You may use AI assistance, but you must:

- record prompts/summary in `docs/test_first_log.md`
- reject out-of-scope code generated by the tool

---

#### <mark style="color:$primary;">4) Add BDD scenarios (Cucumber or Behave)</mark>

Create or update a feature file:

- `bdd/features/lab11.feature`

Minimum:

- 1 feature
- 2 scenarios
  - 1 happy path
  - 1 negative path

If you can run them, great.\
If you cannot run them yet, ensure they are consistent and traceable.

---

#### <mark style="color:$primary;">5) Optional: Selenium (UI automation) — only if justified</mark>

Use Selenium only if all are true:

- you have a stable web UI
- the UI flow is not constantly changing
- you can maintain selectors without wasting time

If you do Selenium:

- automate **one** simple UI scenario only
- keep it minimal (one happy path)

Otherwise, focus on unit/integration + BDD scenario design.

---

#### <mark style="color:$primary;">6) Record evidence (what passed and what you learned)</mark>

In `docs/test_first_log.md`, record:

- test list + outcomes
- what you implemented
- what changed in AC/requirements (if anything)
- what AI generated that you rejected and why

---

### <mark style="color:blue;">Quick guidance:</mark> choose PyTest or JUnit?

- If your prototype slice is easiest in Python: use **PyTest + Behave**
- If your project is Java-oriented: use **JUnit + Cucumber**
- Selenium is optional and should be used only when UI stability justifies it
